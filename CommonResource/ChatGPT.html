std::filesystem은 C++17부터 도입된 표준 라이브러리로, 파일 시스템 작업을 쉽게 수행할 수 있도록 다양한 기능을 제공합니다. 이 라이브러리는 파일과 디렉토리의 경로, 파일 조작, 디렉토리 탐색 등을 직관적으로 다룰 수 있게 해줍니다. 기존에는 이런 작업을 위해 플랫폼에 따라 다른 라이브러리를 사용해야 했지만, std::filesystem은 플랫폼 독립적인 파일 시스템 인터페이스를 제공합니다.

주요 기능

  1. 파일 및 디렉토리 경로 관리 (std::filesystem::path)

    • 파일 경로나 디렉토리를 나타내기 위한 객체로, 경로를 문자열처럼 쉽게 조작하고 관리할 수 있습니다.
    cpp
    std::filesystem::path p = "example/directory/file.txt";

  2. 파일 및 디렉토리 존재 여부 검사 (exists)

    • 특정 경로나 파일이 실제로 존재하는지 확인할 수 있습니다.
    cpp
    if (std::filesystem::exists("example.txt")) {
    // 파일이 존재할 경우 실행할 코드
}

  3. 파일 크기 확인 (file_size)

    • 파일의 크기를 바이트 단위로 가져올 수 있습니다.
    cpp
    std::uintmax_t size = std::filesystem::file_size("example.txt");

  4. 디렉토리 탐색 (directory_iterator)

    • 특정 디렉토리에 있는 파일과 서브 디렉토리를 순회하며 탐색할 수 있습니다.
    cpp
    for (const auto& entry : std::filesystem::directory_iterator("example_directory")) {
    std::cout << entry.path() << std::endl;
}

  5. 파일 복사 및 이동 (copy, rename)

    • 파일을 복사하거나 이름을 바꾸거나 다른 위치로 이동시킬 수 있습니다.
    cpp
    std::filesystem::copy("source.txt", "destination.txt");
std::filesystem::rename("old_name.txt", "new_name.txt");

  6. 파일 삭제 (remove, remove_all)

    • 파일을 삭제하거나, 특정 디렉토리와 그 하위의 모든 파일을 삭제할 수 있습니다.
    cpp
    std::filesystem::remove("example.txt");          // 파일 하나 삭제
std::filesystem::remove_all("example_directory"); // 디렉토리와 하위 모든 파일 삭제

  7. 디렉토리 생성 (create_directory, create_directories)

    • 새로운 디렉토리를 생성합니다. create_directories는 여러 단계의 서브 디렉토리까지 생성할 수 있습니다.
    cpp
    std::filesystem::create_directory("new_directory");
std::filesystem::create_directories("new_directory/sub_directory");

  8. 경로 정보 얻기 (parent_path, filename, extension)

    • 경로의 부모 디렉토리, 파일 이름, 확장자 등 다양한 정보를 얻을 수 있습니다.
    cpp
    std::filesystem::path p = "example/directory/file.txt";
std::cout << "Parent: " << p.parent_path() << std::endl;   // "example/directory"
std::cout << "Filename: " << p.filename() << std::endl;    // "file.txt"
std::cout << "Extension: " << p.extension() << std::endl;  // ".txt"

결론

std::filesystem은 파일 및 디렉토리를 쉽게 다룰 수 있는 도구로, 경로 관리, 파일 검사, 디렉토리 탐색, 파일 조작 등 다양한 작업을 효율적으로 처리할 수 있도록 합니다. 이를 통해 파일 시스템 관련 작업을 코드에서 직관적이고 간단하게 구현할 수 있습니다.

json c++ std::wstring 은 직렬화 역직렬화가 안되네?
ChatGPT의 말:

네, 맞습니다. std::wstring을 JSON으로 직렬화 및 역직렬화하는 것은 조금 더 복잡한 작업입니다. 대부분의 JSON 라이브러리들은 std::wstring을 기본적으로 지원하지 않습니다. 그 이유는 std::wstring이 유니코드로 표현된 문자열이며, JSON은 기본적으로 UTF-8 형식의 std::string을 사용하기 때문입니다. C++에서 자주 사용하는 JSON 라이브러리(예: nlohmann/json, rapidjson)는 주로 std::string을 사용합니다.

해결 방법

std::wstring을 JSON으로 직렬화하거나 역직렬화하려면, std::wstring을 먼저 UTF-8 형식의 std::string으로 변환하고 이를 JSON에 넣는 방식으로 접근해야 합니다.

다음은 std::wstring을 JSON에 사용할 수 있도록 변환하는 방법들입니다:

  1. std::wstring을 std::string으로 변환 (UTF-8 인코딩)
    • 먼저 std::wstring을 UTF-8 형식으로 변환하여 std::string으로 변환한 후, 이를 JSON 직렬화에 사용합니다.
    • 이를 위해 Boost나 C++11 이후의 표준 라이브러리를 사용할 수 있습니다.

다음은 변환 함수의 예시입니다:

cpp
#include <locale>
#include <codecvt>
#include <string>

// wstring -> string (UTF-8로 변환)
std::string WStringToString(const std::wstring& wstr) {
    std::wstring_convert<std::codecvt_utf8<wchar_t>> conv;
    return conv.to_bytes(wstr);
}

// string -> wstring (UTF-8에서 변환)
std::wstring StringToWString(const std::string& str) {
    std::wstring_convert<std::codecvt_utf8<wchar_t>> conv;
    return conv.from_bytes(str);
}

위의 변환 함수를 사용하여 std::wstring을 UTF-8 std::string으로 변환하고 JSON 라이브러리에 전달할 수 있습니다.

  1. nlohmann/json 라이브러리를 사용한 예시

    • nlohmann/json 라이브러리에서 std::wstring을 처리하려면 변환된 UTF-8 문자열을 사용해야 합니다.
    cpp
    #include <nlohmann/json.hpp>
#include <string>
#include <iostream>

using json = nlohmann::json;

int main() {
    std::wstring wide_str = L"안녕하세요, 세계!";

    // wstring을 string으로 변환
    std::string utf8_str = WStringToString(wide_str);

    // JSON으로 직렬화
    json j;
    j["message"] = utf8_str;

    // JSON 출력
    std::cout << j.dump() << std::endl;

    // 역직렬화 후 string을 wstring으로 변환
    std::string deserialized_str = j["message"];
    std::wstring deserialized_wstr = StringToWString(deserialized_str);

    // 결과 출력
    std::wcout << L"역직렬화된 메시지: " << deserialized_wstr << std::endl;

    return 0;
}

  2. 대안으로 UTF-16/UTF-32 사용

    • std::wstring은 플랫폼에 따라 UTF-16 또는 UTF-32를 사용합니다. UTF-16/UTF-32를 그대로 사용하려면 JSON이 이를 다룰 수 있도록 커스텀 로직을 추가해야 하지만, 이는 다소 복잡하고 비효율적일 수 있습니다.
    • 보통의 경우 UTF-8로 변환해서 사용하는 것이 일반적인 해결책입니다.

주의사항

  • std::wstring을 std::string으로 변환할 때는 문자 손실 없이 변환이 이루어져야 하므로 UTF-8로의 변환이 적합합니다.
  • Windows에서는 std::wstring이 UTF-16을 사용하는 반면, Linux에서는 wchar_t가 UTF-32로 구현되어 있습니다. 따라서 플랫폼 간의 호환성도 고려해야 합니다.

이와 같은 방법을 통해 std::wstring을 JSON으로 다루는 것이 가능해집니다. std::wstring은 JSON의 표준 문자열 형식인 UTF-8과 호환되지 않기 때문에 항상 변환 과정이 필요하다는 점을 명심해야 합니다.

다음은 nlohmann::json 라이브러리를 사용하여 std::wstring에 대해 to_json과 from_json을 오버로딩하는 방법을 보여줍니다:

구현 예시

cpp
#include <nlohmann/json.hpp>
#include <string>
#include <codecvt>
#include <locale>

using json = nlohmann::json;

// wstring -> string (UTF-8로 변환)
std::string WStringToString(const std::wstring& wstr) {
    std::wstring_convert<std::codecvt_utf8<wchar_t>> conv;
    return conv.to_bytes(wstr);
}

// string -> wstring (UTF-8에서 변환)
std::wstring StringToWString(const std::string& str) {
    std::wstring_convert<std::codecvt_utf8<wchar_t>> conv;
    return conv.from_bytes(str);
}

// std::wstring에 대한 to_json 오버로딩
void to_json(json& j, const std::wstring& wstr) {
    j = WStringToString(wstr);
}

// std::wstring에 대한 from_json 오버로딩
void from_json(const json& j, std::wstring& wstr) {
    wstr = StringToWString(j.get<std::string>());
}

int main() {
    // 예제 wstring
    std::wstring original_wstr = L"안녕하세요, 세계!";

    // 직렬화: std::wstring -> json
    json j = original_wstr;

    // JSON 출력
    std::cout << j.dump() << std::endl;

    // 역직렬화: json -> std::wstring
    std::wstring deserialized_wstr;
    j.get_to(deserialized_wstr);

    // 결과 출력 (std::wcout은 유니코드 출력을 위해 필요)
    std::wcout << L"역직렬화된 메시지: " << deserialized_wstr << std::endl;

    return 0;
}

코드 설명

  1. WStringToString 및 StringToWString 변환 함수:

    • 이 함수들은 std::wstring과 UTF-8 인코딩의 std::string 간의 변환을 수행합니다. 이를 통해 std::wstring을 JSON에 사용할 수 있는 문자열로 변환할 수 있습니다.

  2. to_json 오버로딩:

    • to_json(json& j, const std::wstring& wstr) 함수는 std::wstring을 UTF-8 std::string으로 변환한 후, 이를 json 객체에 저장합니다.

  3. from_json 오버로딩:

    • from_json(const json& j, std::wstring& wstr) 함수는 JSON 객체에서 std::string을 추출하고, 이를 다시 std::wstring으로 변환하여 원래의 유니코드 데이터를 복원합니다.

  4. 사용 방법:

    • nlohmann::json 라이브러리에서 std::wstring 타입을 바로 직렬화 및 역직렬화할 수 있게 됩니다. json j = original_wstr;와 같이 직렬화하고, j.get_to(deserialized_wstr);와 같이 역직렬화할 수 있습니다.

주의사항

  • UTF-8 인코딩: JSON의 문자열은 기본적으로 UTF-8을 사용하므로, std::wstring을 std::string으로 변환할 때 UTF-8 인코딩을 사용해야 합니다.
  • 플랫폼 호환성: std::wstring은 Windows에서 UTF-16, 다른 플랫폼에서는 UTF-32를 사용하는 경우가 많아, 변환 시 플랫폼 간 호환성에 유의해야 합니다.
  • std::wcout 사용: std::wcout은 std::wstring 출력에 적합한 스트림입니다. 유니코드 문자 출력 시 일반적인 std::cout 대신 std::wcout을 사용해야 제대로 출력할 수 있습니다.

위 방법을 통해 std::wstring 타입을 JSON에서 다룰 수 있게 되고, 직렬화 및 역직렬화가 매우 간단해집니다.

c++ json 이거 enum class도 자동으로 파싱가능?
ChatGPT의 말:

nlohmann/json 라이브러리는 C++에서 열거형 클래스(enum class)를 JSON으로 자동으로 직렬화하거나 역직렬화하지는 않습니다. 기본적으로 enum class는 정수형 타입에 바인딩되지만, JSON 라이브러리는 enum class 값을 다루는 방법을 모르기 때문에 이를 수동으로 처리할 필요가 있습니다.

그러나 nlohmann/json 라이브러리의 to_json 및 from_json 함수를 오버로딩하거나 get과 operator=를 커스터마이징하여 enum class를 JSON으로 직렬화하거나 JSON에서 역직렬화하는 작업을 쉽게 만들 수 있습니다.

구현 방법

  1. enum class 선언:

    • enum class를 정의하고, 각 열거형 값을 문자열로 매핑할 수 있도록 보조적인 로직을 구현합니다.

  2. to_json 및 from_json 함수 오버로딩:

    • JSON과 enum class 간의 변환을 처리하기 위해 to_json과 from_json을 오버로딩합니다.

다음은 enum class를 JSON으로 직렬화 및 역직렬화하는 예시 코드입니다:

cpp
#include <nlohmann/json.hpp>
#include <string>
#include <iostream>

using json = nlohmann::json;

// enum class 정의
enum class Color {
    Red,
    Green,
    Blue
};

// Color -> string 변환을 위한 함수
std::string ColorToString(Color color) {
    switch (color) {
        case Color::Red: return "Red";
        case Color::Green: return "Green";
        case Color::Blue: return "Blue";
        default: return "Unknown";
    }
}

// string -> Color 변환을 위한 함수
Color StringToColor(const std::string& color_str) {
    if (color_str == "Red") return Color::Red;
    if (color_str == "Green") return Color::Green;
    if (color_str == "Blue") return Color::Blue;
    throw std::invalid_argument("Unknown color string");
}

// to_json 함수 오버로딩 (enum class -> json)
void to_json(json& j, const Color& color) {
    j = ColorToString(color);
}

// from_json 함수 오버로딩 (json -> enum class)
void from_json(const json& j, Color& color) {
    color = StringToColor(j.get<std::string>());
}

int main() {
    // Color 열거형 변수
    Color myColor = Color::Green;

    // Color를 JSON으로 직렬화
    json j = myColor;

    // JSON 출력
    std::cout << "Serialized JSON: " << j.dump() << std::endl;

    // JSON에서 Color 역직렬화
    Color newColor;
    j.get_to(newColor);

    // 결과 출력
    if (newColor == Color::Green) {
        std::cout << "Deserialized color is Green" << std::endl;
    } else {
        std::cout << "Deserialized color is not Green" << std::endl;
    }

    return 0;
}

코드 설명

  1. enum class Color 선언:

    • enum class를 사용하여 Color라는 열거형을 정의했습니다. 여기에는 Red, Green, Blue의 세 가지 값이 있습니다.

  2. 보조 함수 작성 (ColorToString 및 StringToColor):

    • ColorToString 함수는 enum class Color 값을 대응되는 문자열로 변환합니다.
    • StringToColor 함수는 문자열을 Color 열거형 값으로 변환합니다.
    • 이 보조 함수들은 to_json 및 from_json을 작성할 때 사용됩니다.

  3. to_json 및 from_json 함수 오버로딩:

    • to_json(json& j, const Color& color) 함수는 enum class 값을 문자열로 변환하여 JSON 객체에 할당합니다.
    • from_json(const json& j, Color& color) 함수는 JSON 문자열을 enum class 값으로 변환합니다.

  4. 사용 예:

    • Color 변수를 JSON으로 직렬화하여 출력하고, 이를 다시 역직렬화하여 Color 값으로 변환하는 과정을 보여줍니다.
    • 직렬화된 JSON 값은 "Green"과 같은 형식으로 출력됩니다.

주의사항

  • 매핑 테이블 사용: 열거형 값이 많을 경우, switch 문 대신 std::map을 사용하여 문자열과 enum 값을 매핑하는 방법을 사용할 수도 있습니다.

매크로를 사용한 구현 예시

  1. 열거형 선언 및 매크로 정의

먼저, 열거형과 매크로를 정의합니다. 이 매크로는 enum class의 각 값을 문자열로 변환하고 역변환할 수 있도록 하는 역할을 합니다.

cpp
#include <nlohmann/json.hpp>
#include <string>
#include <iostream>
#include <unordered_map>
#include <stdexcept>

using json = nlohmann::json;

// enum class 정의
enum class Color {
    Red,
    Green,
    Blue
};

// 매크로 정의: enum class를 문자열과 매핑하는 맵을 생성하고,
// to_json 및 from_json을 정의해줍니다.
#define DEFINE_ENUM_JSON_MAPPING(EnumType, ...)                                     \
    static const std::unordered_map<EnumType, std::string> EnumType##_to_string =   \
        __VA_ARGS__;                                                                \
    static const std::unordered_map<std::string, EnumType> string_to_##EnumType = []{\
        std::unordered_map<std::string, EnumType> m;                                \
        for (const auto& pair : EnumType##_to_string) {                             \
            m[pair.second] = pair.first;                                            \
        }                                                                           \
        return m;                                                                   \
    }();                                                                            \
    inline void to_json(json& j, const EnumType& e) {                               \
        auto it = EnumType##_to_string.find(e);                                     \
        if (it != EnumType##_to_string.end()) {                                     \
            j = it->second;                                                         \
        } else {                                                                    \
            throw std::invalid_argument("Invalid enum value for " #EnumType);       \
        }                                                                           \
    }                                                                               \
    inline void from_json(const json& j, EnumType& e) {                             \
        auto it = string_to_##EnumType.find(j.get<std::string>());                  \
        if (it != string_to_##EnumType.end()) {                                     \
            e = it->second;                                                         \
        } else {                                                                    \
            throw std::invalid_argument("Invalid string value for " #EnumType);     \
        }                                                                           \
    }

// Color enum에 대한 매핑을 정의
DEFINE_ENUM_JSON_MAPPING(Color, {
    { Color::Red, "Red" },
    { Color::Green, "Green" },
    { Color::Blue, "Blue" }
})

int main() {
    // Color 열거형 변수
    Color myColor = Color::Green;

    // Color를 JSON으로 직렬화
    json j = myColor;

    // JSON 출력
    std::cout << "Serialized JSON: " << j.dump() << std::endl;

    // JSON에서 Color 역직렬화
    Color newColor;
    j.get_to(newColor);

    // 결과 출력
    if (newColor == Color::Green) {
        std::cout << "Deserialized color is Green" << std::endl;
    } else {
        std::cout << "Deserialized color is not Green" << std::endl;
    }

    return 0;
}

    • 이 매크로는 특정 enum class와 관련된 JSON 직렬화 및 역직렬화를 쉽게 처리할 수 있도록 매핑 테이블과 to_json/from_json 함수를 자동으로 생성합니다.
    • 매크로는 std::unordered_map 두 개를 정의합니다:
      • 하나는 enum class 값을 문자열로 매핑하기 위한 것이고 (EnumType##_to_string),
      • 다른 하나는 문자열을 enum class 값으로 매핑하기 위한 것입니다 (string_to_##EnumType).
    • 두 개의 unordered_map을 통해 to_json과 from_json이 매핑된 값을 찾고, 이를 이용해 직렬화/역직렬화를 수행합니다.

  2. 사용법:

    • 매크로를 사용하여 열거형 Color에 대한 매핑을 정의했습니다.
    • { Color::Red, "Red" }, { Color::Green, "Green" }, { Color::Blue, "Blue" }와 같은 형식으로 각 열거형 값과 그에 대응되는 문자열을 정의하면 됩니다.

  3. to_json 및 from_json 구현:

    • to_json 함수는 열거형 값을 문자열로 변환하여 JSON 객체에 저장합니다.
    • from_json 함수는 JSON에서 문자열을 가져와 열거형 값으로 변환합니다.
    • 예외 처리를 통해 유효하지 않은 값이 입력되었을 때 적절하게 오류를 발생시킵니다.

Note that in case of multimaps only one key is used in the JSON object and the value depends on the internal order of the STL container. std::map<std::string, int> c_map { {"one", 1}, {"two", 2}, {"three", 3} }; json j_map(c_map); // {"one": 1, "three": 3, "two": 2 } std::unordered_map<const char*, double> c_umap { {"one", 1.2}, {"two", 2.3}, {"three", 3.4} }; json j_umap(c_umap); // {"one": 1.2, "two": 2.3, "three": 3.4} std::multimap<std::string, bool> c_mmap { {"one", true}, {"two", true}, {"three", false}, {"three", true} }; json j_mmap(c_mmap); // only one entry for key "three" is used // maybe {"one": true, "two": true, "three": true} std::unordered_multimap<std::string, bool> c_ummap { {"one", true}, {"two", true}, {"three", false}, {"three", true} }; json j_ummap(c_ummap); // only one entry for key "three" is used // maybe {"one": true, "two": true, "three": true} JSON Pointer and JSON Patch The library supports JSON Pointer (RFC 6901) as alternative means to address structured values. On top of this, JSON Patch (RFC 6902) allows describing differences between two JSON values - effectively allowing patch and diff operations known from Unix. // a JSON value json j_original = R"({ "baz": ["one", "two", "three"], "foo": "bar" })"_json; // access members with a JSON pointer (RFC 6901) j_original["/baz/1"_json_pointer]; // "two" // a JSON patch (RFC 6902) json j_patch = R"([ { "op": "replace", "path": "/baz", "value": "boo" }, { "op": "add", "path": "/hello", "value": ["world"] }, { "op": "remove", "path": "/foo"} ])"_json; // apply the patch json j_result = j_original.patch(j_patch); // { // "baz": "boo", // "hello": ["world"] // } // calculate a JSON patch from two JSON values json::diff(j_result, j_original); // [ // { "op":" replace", "path": "/baz", "value": ["one", "two", "three"] }, // { "op": "remove","path": "/hello" }, // { "op": "add", "path": "/foo", "value": "bar" } // ] JSON Merge Patch The library supports JSON Merge Patch (RFC 7386) as a patch format. Instead of using JSON Pointer (see above) to specify values to be manipulated, it describes the changes using a syntax that closely mimics the document being modified. // a JSON value json j_document = R"({ "a": "b", "c": { "d": "e", "f": "g" } })"_json; // a patch json j_patch = R"({ "a":"z", "c": { "f": null } })"_json; // apply the patch j_document.merge_patch(j_patch); // { // "a": "z", // "c": { // "d": "e" // } // } Implicit conversions Supported types can be implicitly converted to JSON values. It is recommended to NOT USE implicit conversions FROM a JSON value. You can find more details about this recommendation here. You can switch off implicit conversions by defining JSON_USE_IMPLICIT_CONVERSIONS to 0 before including the json.hpp header. Usually, you would do something along those lines: namespace ns { // a simple struct to model a person struct person { std::string name; std::string address; int age; }; } ns::person p = {"Ned Flanders", "744 Evergreen Terrace", 60}; // convert to JSON: copy each value into the JSON object json j; j["name"] = p.name; j["address"] = p.address; j["age"] = p.age; // ... // convert from JSON: copy each value from the JSON object ns::person p { j["name"].template get<std::string>(), j["address"].template get<std::string>(), j["age"].template get<int>() }; It works, but that's quite a lot of boilerplate... Fortunately, there's a better way: // create a person ns::person p {"Ned Flanders", "744 Evergreen Terrace", 60}; // conversion: person -> json json j = p; std::cout << j << std::endl; // {"address":"744 Evergreen Terrace","age":60,"name":"Ned Flanders"} // conversion: json -> person auto p2 = j.template get<ns::person>(); // that's it assert(p == p2); Basic usage To make this work with one of your types, you only need to provide two functions: using json = nlohmann::json; namespace ns { void to_json(json& j, const person& p) { j = json{{"name", p.name}, {"address", p.address}, {"age", p.age}}; } void from_json(const json& j, person& p) { j.at("name").get_to(p.name); j.at("address").get_to(p.address); j.at("age").get_to(p.age); } } // namespace ns That's all! When calling the json constructor with your type, your custom to_json method will be automatically called. Likewise, when calling template get<your_type>() or get_to(your_type&), the from_json method will be called. Some important things: Those methods MUST be in your type's namespace (which can be the global namespace), or the library will not be able to locate them (in this example, they are in namespace ns, where person is defined). Those methods MUST be available (e.g., proper headers must be included) everywhere you use these conversions. Look at issue 1108 for errors that may occur otherwise. When using template get<your_type>(), your_type MUST be DefaultConstructible. (There is a way to bypass this requirement described later.) In function from_json, use function at() to access the object values rather than operator[]. In case a key does not exist, at throws an exception that you can handle, whereas operator[] exhibits undefined behavior. You do not need to add serializers or deserializers for STL types like std::vector: the library already implements these. Simplify your life with macros If you just want to serialize/deserialize some structs, the to_json/from_json functions can be a lot of boilerplate. There are two macros to make your life easier as long as you (1) want to use a JSON object as serialization and (2) want to use the member variable names as object keys in that object: NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE(name, member1, member2, ...) is to be defined inside the namespace of the class/struct to create code for. NLOHMANN_DEFINE_TYPE_INTRUSIVE(name, member1, member2, ...) is to be defined inside the class/struct to create code for. This macro can also access private members. In both macros, the first parameter is the name of the class/struct, and all remaining parameters name the members. Examples The to_json/from_json functions for the person struct above can be created with: namespace ns { NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE(person, name, address, age) } Here is an example with private members, where NLOHMANN_DEFINE_TYPE_INTRUSIVE is needed: namespace ns { class address { private: std::string street; int housenumber; int postcode; public: NLOHMANN_DEFINE_TYPE_INTRUSIVE(address, street, housenumber, postcode) }; } How do I convert third-party types? This requires a bit more advanced technique. But first, let's see how this conversion mechanism works: The library uses JSON Serializers to convert types to json. The default serializer for nlohmann::json is nlohmann::adl_serializer (ADL means Argument-Dependent Lookup). It is implemented like this (simplified): template <typename T> struct adl_serializer { static void to_json(json& j, const T& value) { // calls the "to_json" method in T's namespace } static void from_json(const json& j, T& value) { // same thing, but with the "from_json" method } }; This serializer works fine when you have control over the type's namespace. However, what about boost::optional or std::filesystem::path (C++17)? Hijacking the boost namespace is pretty bad, and it's illegal to add something other than template specializations to std... To solve this, you need to add a specialization of adl_serializer to the nlohmann namespace, here's an example: // partial specialization (full specialization works too) namespace nlohmann { template <typename T> struct adl_serializer<boost::optional<T>> { static void to_json(json& j, const boost::optional<T>& opt) { if (opt == boost::none) { j = nullptr; } else { j = *opt; // this will call adl_serializer<T>::to_json which will // find the free function to_json in T's namespace! } } static void from_json(const json& j, boost::optional<T>& opt) { if (j.is_null()) { opt = boost::none; } else { opt = j.template get<T>(); // same as above, but with // adl_serializer<T>::from_json } } }; } How can I use get() for non-default constructible/non-copyable types? There is a way, if your type is MoveConstructible. You will need to specialize the adl_serializer as well, but with a special from_json overload: struct move_only_type { move_only_type() = delete; move_only_type(int ii): i(ii) {} move_only_type(const move_only_type&) = delete; move_only_type(move_only_type&&) = default; int i; }; namespace nlohmann { template <> struct adl_serializer<move_only_type> { // note: the return type is no longer 'void', and the method only takes // one argument static move_only_type from_json(const json& j) { return {j.template get<int>()}; } // Here's the catch! You must provide a to_json method! Otherwise, you // will not be able to convert move_only_type to json, since you fully // specialized adl_serializer on that type static void to_json(json& j, move_only_type t) { j = t.i; } }; } Can I write my own serializer? (Advanced use) Yes. You might want to take a look at unit-udt.cpp in the test suite, to see a few examples. If you write your own serializer, you'll need to do a few things: use a different basic_json alias than nlohmann::json (the last template parameter of basic_json is the JSONSerializer) use your basic_json alias (or a template parameter) in all your to_json/from_json methods use nlohmann::to_json and nlohmann::from_json when you need ADL Here is an example, without simplifications, that only accepts types with a size <= 32, and uses ADL. // You should use void as a second template argument // if you don't need compile-time checks on T template<typename T, typename SFINAE = typename std::enable_if<sizeof(T) <= 32>::type> struct less_than_32_serializer { template <typename BasicJsonType> static void to_json(BasicJsonType& j, T value) { // we want to use ADL, and call the correct to_json overload using nlohmann::to_json; // this method is called by adl_serializer, // this is where the magic happens to_json(j, value); } template <typename BasicJsonType> static void from_json(const BasicJsonType& j, T& value) { // same thing here using nlohmann::from_json; from_json(j, value); } }; Be very careful when reimplementing your serializer, you can stack overflow if you don't pay attention: template <typename T, void> struct bad_serializer { template <typename BasicJsonType> static void to_json(BasicJsonType& j, const T& value) { // this calls BasicJsonType::json_serializer<T>::to_json(j, value); // if BasicJsonType::json_serializer == bad_serializer ... oops! j = value; } template <typename BasicJsonType> static void to_json(const BasicJsonType& j, T& value) { // this calls BasicJsonType::json_serializer<T>::from_json(j, value); // if BasicJsonType::json_serializer == bad_serializer ... oops! value = j.template get<T>(); // oops! } }; Specializing enum conversion By default, enum values are serialized to JSON as integers. In some cases this could result in undesired behavior. If an enum is modified or re-ordered after data has been serialized to JSON, the later de-serialized JSON data may be undefined or a different enum value than was originally intended. It is possible to more precisely specify how a given enum is mapped to and from JSON as shown below: // example enum type declaration enum TaskState { TS_STOPPED, TS_RUNNING, TS_COMPLETED, TS_INVALID=-1, }; // map TaskState values to JSON as strings NLOHMANN_JSON_SERIALIZE_ENUM( TaskState, { {TS_INVALID, nullptr}, {TS_STOPPED, "stopped"}, {TS_RUNNING, "running"}, {TS_COMPLETED, "completed"}, }) The NLOHMANN_JSON_SERIALIZE_ENUM() macro declares a set of to_json() / from_json() functions for type TaskState while avoiding repetition and boilerplate serialization code. Usage: // enum to JSON as string json j = TS_STOPPED; assert(j == "stopped"); // json string to enum json j3 = "running"; assert(j3.template get<TaskState>() == TS_RUNNING); // undefined json value to enum (where the first map entry above is the default) json jPi = 3.14; assert(jPi.template get<TaskState>() == TS_INVALID ); Just as in Arbitrary Type Conversions above, NLOHMANN_JSON_SERIALIZE_ENUM() MUST be declared in your enum type's namespace (which can be the global namespace), or the library will not be able to locate it, and it will default to integer serialization. It MUST be available (e.g., proper headers must be included) everywhere you use the conversions. Other Important points: When using template get<ENUM_TYPE>(), undefined JSON values will default to the first pair specified in your map. Select this default pair carefully. If an enum or JSON value is specified more than once in your map, the first matching occurrence from the top of the map will be returned when converting to or from JSON. CMake You can also use the nlohmann_json::nlohmann_json interface target in CMake. External To use this library from a CMake project, you can locate it directly with find_package() and use the namespaced imported target from the generated package configuration: # CMakeLists.txt find_package(nlohmann_json 3.2.0 REQUIRED) ... add_library(foo ...) ... target_link_libraries(foo PRIVATE nlohmann_json::nlohmann_json) The package configuration file, nlohmann_jsonConfig.cmake, can be used either from an install tree or directly out of the build tree. Embedded To embed the library directly into an existing CMake project, place the entire source tree in a subdirectory and call add_subdirectory() in your CMakeLists.txt file: # Typically you don't care so much for a third party library's tests to be # run from your own project's code. set(JSON_BuildTests OFF CACHE INTERNAL "") # If you only include this third party in PRIVATE source files, you do not # need to install it when your main project gets installed. # set(JSON_Install OFF CACHE INTERNAL "") # Don't use include(nlohmann_json/CMakeLists.txt) since that carries with it # unintended consequences that will break the build. It's generally # discouraged (although not necessarily well documented as such) to use # include(...) for pulling in other CMake projects anyways. add_subdirectory(nlohmann_json) ... add_library(foo ...) ... target_link_libraries(foo PRIVATE nlohmann_json::nlohmann_json) Embedded (FetchContent) Since CMake v3.11, FetchContent can be used to automatically download a release as a dependency at configure time. Example: include(FetchContent) FetchContent_Declare(json URL https://github.com/nlohmann/json/releases/download/v3.11.3/json.tar.xz) FetchContent_MakeAvailable(json) target_link_libraries(foo PRIVATE nlohmann_json::nlohmann_json) Note: It is recommended to use the URL approach described above which is supported as of version 3.10.0. Supporting Both To allow your project to support either an externally supplied or an embedded JSON library, you can use a pattern akin to the following: # Top level CMakeLists.txt project(FOO) ... option(FOO_USE_EXTERNAL_JSON "Use an external JSON library" OFF) ... add_subdirectory(thirdparty) ... add_library(foo ...) ... # Note that the namespaced target will always be available regardless of the # import method target_link_libraries(foo PRIVATE nlohmann_json::nlohmann_json) # thirdparty/CMakeLists.txt ... if(FOO_USE_EXTERNAL_JSON) find_package(nlohmann_json 3.2.0 REQUIRED) else() set(JSON_BuildTests OFF CACHE INTERNAL "") add_subdirectory(nlohmann_json) endif() ... thirdparty/nlohmann_json is then a complete copy of this source tree. Notes Character encoding The library supports Unicode input as follows: Only UTF-8 encoded input is supported which is the default encoding for JSON according to RFC 8259. std::u16string and std::u32string can be parsed, assuming UTF-16 and UTF-32 encoding, respectively. These encodings are not supported when reading from files or other input containers. Other encodings such as Latin-1 or ISO 8859-1 are not supported and will yield parse or serialization errors. Unicode noncharacters will not be replaced by the library. Invalid surrogates (e.g., incomplete pairs such as \uDEAD) will yield parse errors. The strings stored in the library are UTF-8 encoded. When using the default string type (std::string), note that its length/size functions return the number of stored bytes rather than the number of characters or glyphs. When you store strings with different encodings in the library, calling dump() may throw an exception unless json::error_handler_t::replace or json::error_handler_t::ignore are used as error handlers. To store wide strings (e.g., std::wstring), you need to convert them to a UTF-8 encoded std::string before, see an example. Comments in JSON This library does not support comments by default. It does so for three reasons: Comments are not part of the JSON specification. You may argue that // or /* */ are allowed in JavaScript, but JSON is not JavaScript. This was not an oversight: Douglas Crockford wrote on this in May 2012: I removed comments from JSON because I saw people were using them to hold parsing directives, a practice which would have destroyed interoperability. I know that the lack of comments makes some people sad, but it shouldn't. Suppose you are using JSON to keep configuration files, which you would like to annotate. Go ahead and insert all the comments you like. Then pipe it through JSMin before handing it to your JSON parser. It is dangerous for interoperability if some libraries would add comment support while others don't. Please check The Harmful Consequences of the Robustness Principle on this. However, you can pass set parameter ignore_comments to true in the parse function to ignore // or /* */ comments. Comments will then be treated as whitespace. Order of object keys By default, the library does not preserve the insertion order of object elements. This is standards-compliant, as the JSON standard defines objects as "an unordered collection of zero or more name/value pairs". If you do want to preserve the insertion order, you can try the type nlohmann::ordered_json. Alternatively, you can use a more sophisticated ordered map like tsl::ordered_map (integration) or nlohmann::fifo_map (integration). NLOHMANN_JSON_SERIALIZE_ENUM 매크로

이 매크로는 열거형 값을 문자열로 매핑하거나 역으로 매핑하는 기능을 제공합니다. 매크로는 열거형에 대해 to_json 및 from_json 함수를 자동으로 생성하여 JSON과의 변환을 간편하게 해줍니다.

사용 예시

아래는 열거형을 JSON으로 직렬화하고 역직렬화하는 예시입니다.

cpp
#include <nlohmann/json.hpp>
#include <iostream>
#include <string>

using json = nlohmann::json;

// 열거형 정의
enum class TaskState {
    Stopped,
    Running,
    Completed,
    Invalid = -1
};

// 열거형 직렬화 매크로 정의
NLOHMANN_JSON_SERIALIZE_ENUM(TaskState, {
    {TaskState::Invalid, nullptr},
    {TaskState::Stopped, "stopped"},
    {TaskState::Running, "running"},
    {TaskState::Completed, "completed"}
})

int main() {
    // 열거형 값을 JSON으로 직렬화
    TaskState state = TaskState::Running;
    json j = state;
    
    // JSON 출력
    std::cout << "Serialized JSON: " << j << std::endl; // 출력: "running"

    // JSON을 다시 열거형으로 역직렬화
    json j2 = "completed";
    TaskState newState = j2.get<TaskState>();

    // 결과 출력
    if (newState == TaskState::Completed) {
        std::cout << "Deserialized state is Completed" << std::endl;
    } else {
        std::cout << "Deserialized state is not Completed" << std::endl;
    }

    return 0;
}

코드 설명

  1. TaskState 열거형 정의:

    • TaskState라는 이름으로 Stopped, Running, Completed, Invalid의 네 가지 상태를 갖는 열거형을 정의합니다.

  2. NLOHMANN_JSON_SERIALIZE_ENUM 매크로 사용:

    • 이 매크로는 TaskState 값을 JSON 문자열로 변환할 수 있는 매핑을 정의합니다.
    • {TaskState::Invalid, nullptr}과 같이 열거형 값 Invalid를 nullptr에 매핑하면, JSON에서 해당 상태를 나타낼 수 없는 경우 기본적으로 Invalid 상태로 취급하게 됩니다.
    • 다른 열거형 값들은 각각 "stopped", "running", "completed" 등의 문자열로 매핑됩니다.

  3. 직렬화 및 역직렬화:

    • json j = state;를 통해 TaskState::Running을 JSON 문자열 "running"으로 직렬화합니다.
    • j2.get<TaskState>()를 통해 JSON 문자열 "completed"을 TaskState 열거형 값으로 역직렬화합니다.

주의사항

  • 매크로 위치: NLOHMANN_JSON_SERIALIZE_ENUM() 매크로는 해당 열거형이 정의된 네임스페이스 안에 선언해야 합니다. 그래야 라이브러리에서 이를 제대로 찾을 수 있습니다.
  • 역직렬화 기본값: get<ENUM_TYPE>() 함수는 JSON 값이 정의되지 않은 경우 매크로에서 첫 번째 매핑된 값을 기본으로 사용합니다. 따라서 기본값을 신중하게 설정해야 합니다.
